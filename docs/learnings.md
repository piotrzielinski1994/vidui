- `nvm` on this machine is shimmed to print "use `mise` instead" and does nothing; node is managed by mise (`.nvmrc` pins 24, mise has 24.17.0). In non-interactive bash, activate first: `eval "$(mise activate bash)"` then `mise exec -- <cmd>` to get node 24.
- TanStack Hotkeys: `@tanstack/react-hotkeys` provides the React `useHotkeys` + `HotkeysProvider` (the framework-agnostic core is the separate `@tanstack/hotkeys`). Hotkey strings are case-sensitive uppercase, e.g. `"Mod+K"`. Under jsdom the lib resolves `Mod` to `Control`, so hotkey tests fire `{Control>}k{/Control}`, not Meta. Root cause: `detectPlatform()` checks UA for "mac" then "win"; jsdom's UA is `Mozilla/5.0 (darwin) ...` and "da**rwin**" contains "win" -> resolves to `"windows"`. Production "Mod+K" still maps to Cmd on real macOS - this is a test-env quirk only.
- shadcn Button keeps `react-refresh/only-export-components` as an accepted lint *warning* (the canonical upstream file exports `buttonVariants` alongside the component). Lint exits 0 with that one warning.
- Vite dev server uses `strictPort: true` on **1432** (Tauri requirement; HMR fallback 1433). Moved off 1420 specifically because the sibling `requi` repo dev-serves on 1420 - same-port clashes meant a headless screenshot of localhost:1420 could silently capture requi's app instead. Both the vite `server.port` and tauri.conf.json `devUrl` must agree. Also: a `npm run dev` started with `&` in a Bash tool call dies when that call's shell exits - launch it with the tool's `run_in_background` instead so it survives across calls.
- `cmdk` (command palette) calls `Element.prototype.scrollIntoView` on mount to keep the highlighted item visible; jsdom doesn't implement it, so any test rendering a `Command`/`CommandDialog` throws `TypeError: e.scrollIntoView is not a function`. Fix is a one-line no-op polyfill in `src/test/setup.ts` (next to the `ResizeObserver` stub). cmdk filters/highlights rows itself - assert on `role="option"` rows + the `CommandEmpty` text. Also: in an integration test the sidebar list AND the viewport both render the active video's name, so scope name assertions to the viewport region (`within(getByRole("region", { name: /video viewport/i }))`) to avoid "found multiple elements".
- Tauri asset protocol needs BOTH config and a Cargo feature: enabling `app.security.assetProtocol` in `tauri.conf.json` is not enough - the `tauri` crate must also carry `features = ["protocol-asset"]`, or `cargo build` fails with "The `tauri` dependency features ... does not match the allowlist defined under `tauri.conf.json` ... add the `protocol-asset` feature". `convertFileSrc` then resolves local files for `<video src>`.
- jsdom has no real media engine: `HTMLMediaElement.prototype.play/pause/load` throw "Not implemented", so any test that mounts a real `<video>` needs no-op stubs in `src/test/setup.ts` (`play` returns a resolved promise). Also `currentTime`'s setter throws - back it with a plain field via `Object.defineProperty` so seek code (`video.currentTime = sec`) is observable. And `Element.prototype.setPointerCapture/releasePointerCapture/hasPointerCapture` throw - stub them (assign directly; a `if (!...)` guard won't override the throwing built-in) for any pointer-drag UI (the seek slider). Real playback can only be verified manually via `npm start`; tests assert observable state (src attribute, isPlaying-driven icon, aria-* on the slider) and mocked IPC calls, never pixels. The single mockable IPC seam is `@/lib/tauri` - mock that, never the components.
- The eslint `react-hooks` rule (this repo's config) flags a SYNCHRONOUS `setState` in an effect body as an error ("Calling setState synchronously within an effect can trigger cascading renders"). Bit us adding the 5s title-overlay timer: `useEffect(() => { setVisible(true); setTimeout(() => setVisible(false), 5000) }, [id])` errors on the `setVisible(true)`. Fix pattern: don't write the "reset to shown" - instead key visibility off the changing id. Use `titleHiddenForId` state (only the timer writes it, async = allowed) and render `visible = titleHiddenForId !== activeVideo.id`; a new id is shown by default, no synchronous setState needed.
- The `<video>` element must NEVER be conditionally unmounted/remounted, or playback restarts from 0. Bug that bit us: `WorkspaceLayout` returned two different parent trees (a bare `<div>` when the sidebar was hidden vs a `<ResizablePanelGroup>` when shown), so toggling the sidebar remounted `Content` -> the `<video>` -> restart. Fix: one stable tree where the content `<ResizablePanel key="content">` is always mounted and only the sidebar panel/handle are conditionally rendered. Guard this with a test asserting DOM-node identity survives the toggle: `expect(container.querySelector("video")).toBe(before)`. (Also: `react-resizable-panels` in this repo does NOT accept an `order` prop - a stable `key` is enough to preserve the panel across re-renders.)
- Native window fullscreen (Tauri `setFullscreen`) only resizes the OS window - it does NOT hide app chrome. For an immersive view, track an `isFullscreen` flag in `WorkspaceProvider` (set from `toggleFullscreen()`'s returned new state) and gate the sidebar/transport/overlay on it. `toggleFullscreen` returns the new boolean so JS state stays in sync with the window. But that only covers OUR toggle - fullscreen entered/left outside the app (F11, OS Esc, green button) bypasses it and leaves `isFullscreen` stale. Fix: `watchFullscreen(onChange)` subscribes via `getCurrentWindow().onResized` and re-queries `isFullscreen()` on every transition; `Workspace` wires it to `setFullscreen` so the real window state is the single source of truth. The store verb `setFullscreen` is the raw `useState` setter (reference-stable) so the subscribing effect doesn't re-subscribe on every render.
- `getCurrentWindow()` (from `@tauri-apps/api/window`) THROWS outside a Tauri host ("Cannot read properties of undefined (reading 'metadata')") - so it breaks plain-browser `npm run dev` AND any jsdom test that renders the real app without mocking `@/lib/tauri`. Any wrapper using it (e.g. `watchFullscreen`) must `try/catch` and fail safe (return a no-op unlisten), not assume a host. The e2e bootstrap test only mocks `@tauri-apps/api/core`, not `/window`, so it exercises this path.
- Switching the active `<video>` while playing: drive playback from the element's `onLoadedData` (guarded by `isPlaying`) for the freshly-loaded source, and keep the effect's imperative `play()` for resume/pause of the already-loaded element. Calling `play()` synchronously after changing `src` races the load and rejects. Test by firing `fireEvent.loadedData(video)` and asserting `video.paused === false` (back `paused`/`play`/`pause` with a `_paused` field in the jsdom stub).
- UNIVERSAL PLAYBACK = bundled ffmpeg, NOT the webview `<video>` codecs and NOT embedded mpv. The OS webview can't decode every format (macOS WKWebView: no file-based WebM/VP9, no MKV/AVI, no AV1; webkit.org #216652). Embedding mpv (render-API under a transparent webview) was tried and ABANDONED - transparency broke the whole macOS window and the plugin is macOS-untested. The shipping approach: `prepare_media` (Rust, `src-tauri/src/media.rs`) probes codecs via ffprobe; H.264/AAC plays as-is, anything else is transcoded to a **fragmented** MP4 (`-movflags frag_keyframe+empty_moov+default_base_moof`, `h264_videotoolbox` on macOS) and the path is returned the instant the first ~256KB lands. A fragmented MP4 is progressively playable, so `<video>` starts in <1s and streams while ffmpeg races ahead (~15x realtime; AV1 decode is the bottleneck, still far faster than playback). ffmpeg/ffprobe are now BUNDLED static binaries (Tauri sidecars), not system PATH - see the sidecar learning below.
- macOS WKWebView LOSES first-responder after a fullscreen transition (and on launch) - keyboard events stop reaching the webview until the user physically clicks the window. This is tao#208 / tauri#9389, NOT fixable from JS: `getCurrentWindow().setFocus()` focuses the window, not the inner web content view, and `el.focus()` on a DOM node does nothing for it. The ONLY fix is native: call `makeFirstResponder:` on the WKWebView's `NSView` in Rust. See `src-tauri/src/focus.rs` (`focus_webview` command, objc2 - `window.with_webview(|w| msg_send![w.ns_window(), makeFirstResponder: w.inner()])`). Invoke it on every fullscreen transition (in `watchFullscreen`'s onResized) AND every window focus-gain (`onFocusChanged`). Burned ~hours here treating it as DOM/window focus - it is neither.
- KNOWN ENGINE BUG (not app-fixable): on macOS WKWebView/WebKit audio can restart ~0.5-1s after video when resuming H.264 from pause (webkit.org #293544, #221334). In the media pipeline, not the app. Don't chase it in JS.
- Viewport single-click = play/pause, double-click = fullscreen on the SAME element. DON'T debounce the click behind a setTimeout to "wait out" the dblclick - the ~200ms delay is perceptible and feels laggy vs the instant transport button. Instead toggle SYNCHRONOUSLY in `onClick`: a double-click fires `click` twice (toggle + toggle-back = net no change) then `dblclick` -> fullscreen lands on top. Snappy, no stray toggle. The `dblClick -> isPlaying unchanged` test still passes because the two toggles cancel.
- jsdom natively backs `HTMLMediaElement` `volume` / `muted` / `playbackRate` (getters AND setters) - unlike `currentTime`/`play`/`pause`, these need NO stub in `src/test/setup.ts`. So volume/mute/speed tests can assert `video.volume` / `video.muted` / `video.playbackRate` directly on the real element. Float gotcha: stepping volume by `0.05` accumulates IEEE drift (`1 - 0.05 - 0.05 = 0.8999...`), so `changeVolume`/`setVolume` round to 2 decimals (`Math.round(v*100)/100`); playback rate rounds to 1 decimal in `clampRate`. Both keep the on-screen readout/aria clean without a separate format step.
- ffmpeg/ffprobe ship as Tauri **sidecars** (`externalBin` in `tauri.conf.json` = `["binaries/ffmpeg","binaries/ffprobe"]`), resolved in Rust via `app.shell().sidecar("ffmpeg")` (the `tauri-plugin-shell` crate + `shell:allow-execute` capability with `sidecar:true, args:true`). Bare names only - the plugin appends the `-<TARGET_TRIPLE>[.exe]` suffix and picks dev-vs-bundle dir, so DON'T hand-roll `current_exe()` path-joining. `cargo build` HARD-FAILS ("resource path `binaries/ffmpeg-<triple>` doesn't exist") if the binary for the host triple is missing, so `scripts/fetch-ffmpeg.sh` must run before any build. Binaries are gitignored (`src-tauri/binaries/`), fetched + SHA-256-pinned per triple. macOS = GPLv3 (no off-the-shelf LGPL static macOS build exists), Windows = LGPLv3 - see adr.md.
- `tauri-plugin-shell` `.spawn()` (unlike bare `std::process::Command`) force-pipes stdout/stderr into a bounded `channel(1)`; if you don't drain the returned receiver, ffmpeg BLOCKS once the OS pipe buffer fills, stalling a background/streaming encode. Drain it in a detached `async_runtime::spawn` task. Conversely, the detach for streaming WORKS because `CommandChild` has NO `impl Drop` and `kill()` takes `self` by value - dropping the child does not kill ffmpeg, so it keeps encoding the fragmented MP4 after `prepare_media` returns. Read termination via `CommandEvent::Terminated` in the drain task (the plugin owns the child; there's no `try_wait`).
- Radix `DropdownMenu` is modal: while open it sets `aria-hidden` on the trigger, so a test that re-opens the menu between selections only works if the menu CLOSES on each select (the default). Don't `preventDefault` on `onSelect` of a `DropdownMenuCheckboxItem` if tests rely on reopening - keep the close-on-select default and let `onCheckedChange` toggle state. Multi-select is achieved by reopening, not by keeping the menu open.
- A context verb consumed inside a `useEffect` dependency array (e.g. `Workspace` subscribing to `watchFileDrop(addVideos)`) MUST have a stable identity, or the effect tears down + re-subscribes on every state change that rebuilds the context value memo. With an event-subscription seam mocked as `watchFileDrop = vi.fn(h => { handler = h; return Promise.resolve(() => { handler = null }) })`, a re-subscribe runs the PREVIOUS cleanup which nulls the captured `handler`, so a later fired drop event hits nobody and the test sees "1 row, expected 2". Fix: build the verb with `useCallback([], ...)` and read live state via refs (`sourceVideosRef`/`activeVideoIdRef` mirrored in an effect) - the exact same reference-stability pattern already used for `setFullscreen`. Symptom in tests was specifically: click-a-row-to-start-playback (which mutates context) between mount and the drop silently killed the subscription.
- `getCurrentWebview().onDragDropEvent` (from `@tauri-apps/api/webview`) is the Tauri 2 drag-drop seam; like `getCurrentWindow()` it THROWS outside a Tauri host, so wrap it in try/catch returning a NO_UNLISTEN (same guard as `watchFullscreen`). Its payload union is `{type:'enter'|'over'|'drop', paths, position} | {type:'leave'}`; we flatten to `enter|leave|drop` (dropping `over`). Native drag-drop is enabled by default (`dragDropEnabled`), no config/permission change needed - `core:default` covers the event `listen`. Folder recursion + extension filtering belongs in Rust (`expand_dropped_paths`), not JS: the webview hands a mixed file+folder path list and expanding dirs in JS would need a second IPC round-trip per dir.
- SUPERSEDES the "fragmented MP4 / starts in <1s" approach above: serving a STILL-GROWING file via the asset protocol broke duration. `<video src="asset://...">` reads the served file's byte length ONCE (at request time); a fragmented MP4 that ffmpeg is still appending to makes the element treat the buffered ~1s as the whole clip -> transport shows `00:01 / 00:01` for a long video (VLC was fine because it reads the finished SOURCE file, not our temp file). Fix: `prepare_media` now writes a COMPLETE, finalized MP4 (`-movflags +faststart`, NOT `frag_keyframe+empty_moov`), AWAITS `CommandEvent::Terminated`, and atomically renames `<hash>.mp4.part` -> `<hash>.mp4` (rename = the cache-complete marker, so `target.exists()` alone means "done"). Trade-off: lost the <1s start; a true re-encode shows the spinner for its full run. Because we now await the encode to completion, the detached drain-task + no-Drop-kill trick is gone - we just `rx.recv()` the bounded channel to the end inline (which also yields the exit code) and let the child drop normally.
- Repeat-one (replay-in-place) needs the SEEK effect to resume playback, not the play effect. `reportEnded` for repeat-one sets `seekToSec=0` + `setIsPlaying(true)`, but the video was ALREADY playing (`isPlaying` true), so the play-effect (`[isPlaying, sourceForActive]`) doesn't re-fire and the element - which a real browser PAUSES when it emits `ended` - stays paused at 0. Symptom on-device: video ends, doesn't replay; manual pause+play (which flips `isPlaying`) revives it. Fix: the seek effect resumes `play()` when `isPlaying && element.paused` after setting `currentTime`, and depends on `[seekToSec, isPlaying]`. jsdom does NOT auto-pause on `ended` (no real engine), so the failing test must `video.pause()` before `fireEvent.ended(video)` to reproduce, AND must start playback first (toggle-play) so `isPlaying` is already true - else the false->true flip masks the bug.
- `SettingsProvider` (mirrored from `requi`) renders `null` until its async `store.load()` resolves - to avoid mounting `WorkspaceProvider` with default `initial*` props before the persisted values arrive (the `initial*` props are init-only `useState` initializers, so they must be correct on first mount or the persisted setting is lost). Consequence for tests: any test rendering `<Workspace/>` under `SettingsProvider` (even with the in-memory store, whose `load()` is an already-resolved promise) sees an EMPTY body on the first synchronous tick. So the FIRST query after render must be an `await screen.findBy*` (or `waitFor`), never a synchronous `getBy*`/`expect(...).toBeInTheDocument()` - else it asserts against `<body><div/></body>` and fails. Bit the 4 `<Workspace/>` tests (palette/open-files/drop/extended-transport): their leading "precondition" assertions had to become `findBy`.
- shadcn's `Switch` ships `rounded-full` on both the track and the thumb - against design.md's no-rounded-corners rule. The vidui `ui/switch.tsx` strips ALL rounding (square track + square thumb) and imports radix via the repo's `radix-ui` named-namespace convention (`import { Switch as SwitchPrimitive } from "radix-ui"`, same as `dropdown-menu.tsx`), NOT the per-package `@radix-ui/react-switch`. Treat any shadcn primitive's `rounded-*` as a defect to strip on add.
- Reveal-transport-on-hover lives in `Content` as LOCAL state reading `useSettings().revealTransportOnHover` directly - NOT workspace state. The hidden bar renders as an `absolute inset-x-0 bottom-0` overlay over the `<video>` (no layout shift, video keeps its size); the docked bar is still the normal-flow `{isTransportVisible && <TransportBar/>}`. Both render the same `<TransportBar>` - that's fine, only one is in the tree at a time (hidden-and-revealed XOR docked). The reveal is driven by `onMouseMove` (NOT enter/leave): each move shows the bar + restarts an `IDLE_HIDE_MS` (3000ms) `setTimeout`; the timer hides it after idle; `onMouseLeave` clears the timer + hides immediately. Test it with `vi.useFakeTimers()` + `fireEvent.mouseMove/mouseLeave` + `vi.advanceTimersByTime`; under fake timers, first `await act(async()=>{await Promise.resolve()})` so the SettingsProvider's async load (null-gate) resolves before asserting. GOTCHA: the overlay bar is a CHILD of the move-tracked wrapper, so mouse moves over the bar BUBBLE to the wrapper's `onMouseMove` and would re-arm (then fire) the hide timer even while the cursor rests on the controls. Fix: an `isOverBar` ref set by the bar's own `onMouseEnter`/`onMouseLeave` (clear timer on enter, restart on leave), and `handleMouseMove` early-returns when `isOverBar.current` so bubbled moves don't re-arm. Use a ref (not state) so the move handler doesn't stale-close over it and there's no extra render.
- cmdk filters items by their `value` + an optional `keywords` string array ONLY - the visible label text and description are NOT searched beyond `value`. So an action whose user-spoken name differs from its label (e.g. "bottom bar" vs "Toggle transport bar") won't match unless you pass `keywords`. The registry's `ShortcutAction.keywords?: string[]` is threaded `action.keywords -> PaletteCommand.keywords -> <CommandItem keywords={...}>`. When adding an action, add aliases users might type.
- Fullscreen is the canonical "lives outside the registry" trap: it was double-click-only, so it was absent from the palette AND unbindable. Promoting any such gesture-only action to a `SHORTCUT_ACTIONS` entry + a `workspace.tsx` handler gives it palette + hotkey + rebind for free. Audit: every user-facing verb should have a registry entry unless there's a reason not to.
- Persistence is wired WITHOUT coupling `WorkspaceProvider` to the settings store: the provider stays DI-pure (optional `initial*` seed props + optional `on*Change` callbacks, mirroring the existing `initialSortKeys` seam and `requi`'s `onTreeChange`); the route-level bridge (`routes/index.tsx`) reads `useSettings()`, seeds `initial*` from the loaded settings, and maps the `on*Change` callbacks to the savers. This kept all ~13 existing provider-only tests untouched - only the 4 that render the full `<Workspace/>` (which calls `useActionHotkeys` -> `useSettings`) needed a `SettingsProvider` wrapper.
- `@tanstack/hotkeys` `validateHotkey("Mod+,")` is valid (comma as a literal key); `"Mod+Comma"` validates too but emits an "Unknown key" warning - `safeNormalize` rejects anything with that warning, so use the literal `,` for the open-settings binding, not `Comma`.
- Punctuation hotkeys: the literal `,` and `.` both `validateHotkey` clean (valid, NO "Unknown key" warning), so `safeNormalize` accepts them as-is - same as `[`/`]`. Use the literal char, not `Comma`/`Period` (those validate but emit the "Unknown key" warning that `safeNormalize` rejects, mirroring the `Mod+,` vs `Mod+Comma` rule). No collision risk between `,` (frame-step-back) and `Mod+,` (open-settings) - the modifier makes them distinct hotkeys.
- More clean hotkey tokens (extends the `,`/`.`/`[`/`]` catalog): `=`, `-`, `Mod+0`, `Mod+Shift+R` all `validateHotkey` without an "Unknown key" warning, so `safeNormalize` accepts them as-is for the viewport-transform binds. Use the literal `=`/`-`, not `Equal`/`Minus`/`Plus`.
- Viewport transforms (FR-10) are pure CSS on the `<video>`: inline `object-fit` (contain/cover/fill) + `transform: rotate(Ndeg) scale(Z)` + `transform-origin: center`, with the element sized `h-full w-full` (NOT `max-h/w-full`) so `object-fit` is meaningful and `contain` reproduces the old letterbox. The viewport region needs `overflow-hidden` so a zoomed video doesn't bleed. State is one session-sticky `viewportTransform` context field, deliberately NOT touched by `activateVideo` (persists across switches) and NOT written to the settings store (stays out of FR-7). Pure cycle/clamp/format logic lives in `viewport-transform.ts` (ADT helpers), keeping the JSX ifology-free. Known limit: pure CSS `rotate` does not refit a 90/270 landscape clip - it letterboxes until the user zooms; auto-refit was out of scope.
- Test gotcha (instance of the "found multiple elements" trap): an integration `Probe` that renders each transform facet as its own `<output>` (e.g. `cover`) collides with the transport readout's combined string (`90deg cover 1.5x`), so `getByText(/cover/i)` finds two. Assert the full combined readout string (`getByText("90deg cover 1.5x")`) instead of per-facet regexes when a probe mirrors the same words.
- cmdk (command palette) hijacks selection to the row under the mouse: ArrowDown scrolls the list via `scrollIntoView`, sliding a NEW row beneath the stationary cursor, which fires that row's `onPointerMove` -> cmdk reselects the hovered row. On-device symptom: from the last visible row, ArrowDown scrolls one item but the highlight SNAPS BACK to whatever row landed under the cursor (often the first, "Open files"). Fix: pass `disablePointerSelection` to the cmdk `Command` root (`src/components/ui/command.tsx`) so arrow keys stay authoritative; clicks still run a row (they use `onClick`, not pointermove). Reproduce in jsdom WITHOUT layout by `fireEvent.pointerMove` on a non-selected row and asserting the keyboard selection is unchanged - no `scrollIntoView`/offsetHeight needed. Gotcha reading the selected row: `[cmdk-item][aria-selected="true"]`'s `textContent` includes the shortcut span's glyph (e.g. trailing `␣` for Space), so query the row's inner label `span`, not the item.
- WKWebView ships browser defaults the app must suppress: right-click shows a native media context menu (Pause/Loop/Enter PiP/Inspect Element/Show Media Statistics) on `<video>`, and browser-reserved combos (Cmd+R reload, Cmd+=/-/0 zoom, Cmd+F/G find, Cmd+P print, Cmd+S save, Cmd+U view-source) fire native actions. Fix is frontend-only (no Rust): `installBrowserDefaultGuards(window)` in `src/lib/browser-defaults.ts`, mounted once via a `useEffect` in `AppProviders`. It `preventDefault()`s `contextmenu` app-wide and `preventDefault()`s keydown for `isReservedBrowserShortcut` (a pure fn: primary modifier (meta OR ctrl) + a reserved key, lowercased). KEEP text-edit (copy/paste/cut/select-all/undo/redo) + OS combos (quit/close/minimize/hide) working - they're deliberately NOT in the reserved set. CRUCIAL: do NOT `stopPropagation()` - `Mod+0` is both the browser zoom-reset AND the app's `reset-viewport` hotkey; `preventDefault` blocks only the native zoom while the TanStack `useHotkeys` window listener still fires, so the app action survives. Same holds for any future app hotkey that overlaps a reserved combo.
- ffprobe `is_directly_playable` checking only the CODEC (h264/aac) was a bug: an MKV with h264/aac was tagged "direct" and served raw, but macOS WKWebView cannot play MKV/AVI containers at all. The decision must be CONTAINER-AWARE. Now a pure `plan_media(container, vcodec, acodec) -> MediaPlan` ADT (`Passthrough` | `Convert{VideoAction, AudioAction}`): MP4-family container (`format_name` contains "mp4") + h264 + aac/mp3/none -> passthrough; else convert, copying streams that are already fine (`-c:v copy` remux is near-instant) and re-encoding only those that aren't. Probe the container via ffprobe `-show_entries format=format_name`. All branching is in the pure fn (unit-tested per container/codec combo); the ffmpeg spawn path just matches on the plan - no codec ifology there.
- HLS playback has two follow-on gotchas. (1) DURATION: an EVENT playlist declares no total length until `#EXT-X-ENDLIST`, so `<video>.duration` reads `Infinity` (and `NaN` before metadata) mid-stream -> transport showed `Infinity:NaN`. Fix: ffprobe already gives `format.duration`; carry it through `PreparedMedia.durationSec` to the FE and use it whenever `element.duration` isn't finite/positive (`resolveDuration` in viewport.tsx). (2) COLD START: the first dropped file took ~3s even though standalone ffprobe+ffmpeg-to-first-segment is ~156ms - the gap is the FIRST spawn of the bundled ~60MB Developer-ID-signed sidecars (macOS one-time Gatekeeper check + page-in). Fix: `prewarm_sidecars` runs `ffprobe -version` + `ffmpeg -version` (detached) at app setup, behind the empty UI, so the user's first real prepare is warm. GOP also matters: with `-c:v copy` HLS can only cut segments at keyframes (this source had a 5.3s GOP), so the first segment is a whole GOP - not the dominant cost here but worth knowing if start latency needs more shaving.
- Instant playback ("like VLC") for non-MP4 files = HLS streaming, NOT pre-transcode. Measured: remuxing video+audio of a 32-min file = 0.46s (video is `-c:v copy`); the only slow part was encoding audio of the WHOLE file up front (~11s) even though the encoder runs ~175x realtime. So don't pre-encode - stream. ffmpeg writes an EVENT `.m3u8` + TS segments progressively (`-f hls -hls_time 4 -hls_playlist_type event`, NO `append_list` - that flag injects a stray `#EXT-X-DISCONTINUITY` and is pointless on a fresh dir); first segment lands in ~88ms, `<video>` starts then, encoder races ahead. HARD CONSTRAINT: WKWebView plays HLS via AVFoundation, which BYPASSES Tauri custom-scheme handlers (`asset://`) - so HLS will NOT load over `asset://`, it needs a real `http://localhost`. Hence a loopback `tiny_http` server (bind `127.0.0.1:0`, serve only the HLS temp root, `application/vnd.apple.mpegurl` for .m3u8 + `video/mp2t` for .ts, path-traversal guard). `CommandChild` has no `Drop`, so the ffmpeg child keeps running after `prepare_media` returns (needed for stream-ahead); drain its event channel in a detached `async_runtime::spawn` or the bounded pipe stalls the encode. Passthrough (mp4/h264/aac) stays on `asset://` (already 196ms, range-capable). No persistent cache - per-session temp dir, killed+cleaned on file switch, wiped at startup. tiny_http's `request.url()` does NOT percent-decode, so `%2e%2e` traversal arrives as one literal component and fails canonicalize anyway - but the `Component::Normal`-only guard + canonical-containment `starts_with(root)` check is the real defense (also defeats symlink escape).
- Two transcode bugs surfaced by FR-13's `v2` cache bump (both latent before, masked because nothing re-exercised a cache MISS): (1) ffmpeg writes to `<hash>.mp4.part` (atomic-rename temp) and infers the muxer from the EXTENSION - `.part` is unknown, so it exits 234 `Unable to choose an output format`. Fix: pass `-f mp4` explicitly. Lesson: any ffmpeg output to a non-standard extension MUST force `-f`. (2) The default `aac` encoder uses the two-loop bit allocator, ~4x slower than needed: a 32-min opus->aac re-encode took 44s (audio encode alone = 44s; opus decode = 3s; video was `-c:v copy`, instant). `-aac_coder fast` cuts it to 11s at transparent playback quality. `aac_at` (macOS AudioToolbox) was slower (13-21s) and platform-locked, so `-aac_coder fast` wins cross-platform. Also: capture ffmpeg `stderr` (the shell plugin pipes it as `CommandEvent::Stderr`) into the failure log - exit code alone is blind; `-v error` keeps it to real diagnostics.
- Slow drop->playback (FR-13) had two fixable wastes, both in `prepare_media`, neither in transcode cost. (1) **Cache check must precede probing.** `cache_path` depends only on the source path, so on a cache HIT the codec probe is pure waste - reorder so `target.exists()` returns early BEFORE any ffprobe. (2) **One ffprobe spawn, not three.** Each Tauri sidecar spawn carries ~500ms overhead though bare ffprobe is ~50ms; three sequential spawns (v/a/container) cost ~1.6s. Collapse to one `ffprobe -show_entries format=format_name:stream=codec_name,codec_type -of json` feeding a pure `parse_probe_json(&str) -> ProbeResult` (total `serde_json` parse: `.unwrap_or(Null)` + `Value` index + `.as_str().unwrap_or("")` -> panic-proof on junk/empty; first video/audio stream wins). (3) **The asset-protocol `<video>` load was slow (5.5s) because the CACHED file was a stale FRAGMENTED MP4** (moov ~1KB empty sample table + thousands of moof/mdat + trailing mfra) written before the faststart fix - `target.exists()` kept serving it forever. A fragmented MP4 forces WKWebView to walk every fragment over the 1MB-capped asset protocol to build a seekable timeline. Fix: VERSION the cache dir (`vidui-transcode/v2/<hash>.mp4`) so any namespace bump invalidates all pre-faststart files at once - cheaper + deterministic than runtime atom-sniffing. Lesson: when changing the bytes a cache stores, bump a namespace segment in the cache key, else `exists()` serves the old format indefinitely.
- The app had ZERO logging until FR-12 (no `log::`/`println!`; `tauri dev` shows only build output), which made on-device playback verification guesswork - couldn't tell an 8s MKV load from a cache hit. Added `tauri-plugin-log` (Tauri 2: `Target::new(TargetKind::LogDir { file_name: Some(..) })`, the plugin appends `.log`). "New file per launch" needs NO rotation logic: compute `vidui-<YYYYMMDDHHMMSS>` once at startup and pass it as the target's `file_name` - a new process gives a new timestamp gives a new file. GOTCHA: the plugin's defaults are `max_file_size=40_000` bytes + `RotationStrategy::KeepOne`, so a real session would rotate and DELETE mid-run; set `KeepAll` + a large `max_file_size` to keep a whole launch in one file. Local-time stamp (not the plugin's UTC default) to match the `docs/features/*` convention - sourced via `chrono::Local::now()` (added `chrono` with only the `clock` feature), NOT `time::now_local()` which bails to UTC under multithreading. The pure formatter `launch_log_name(y,mo,d,h,mi,s)` is the only unit-tested part; the clock-reading `current_launch_log_name()` wrapper delegates to it. No capability/permission change needed because logging is Rust-side only (no `Webview` target, no JS log command). `prepare_media` is instrumented at INFO (path/container/codecs, `plan={:?}`, cache HIT/MISS, elapsed ms) + ERROR on probe/ffmpeg failure.
