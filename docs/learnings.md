- `nvm` on this machine is shimmed to print "use `mise` instead" and does nothing; node is managed by mise (`.nvmrc` pins 24, mise has 24.17.0). In non-interactive bash, activate first: `eval "$(mise activate bash)"` then `mise exec -- <cmd>` to get node 24.
- TanStack Hotkeys: `@tanstack/react-hotkeys` provides the React `useHotkeys` + `HotkeysProvider` (the framework-agnostic core is the separate `@tanstack/hotkeys`). Hotkey strings are case-sensitive uppercase, e.g. `"Mod+K"`. Under jsdom the lib resolves `Mod` to `Control`, so hotkey tests fire `{Control>}k{/Control}`, not Meta. Root cause: `detectPlatform()` checks UA for "mac" then "win"; jsdom's UA is `Mozilla/5.0 (darwin) ...` and "da**rwin**" contains "win" -> resolves to `"windows"`. Production "Mod+K" still maps to Cmd on real macOS - this is a test-env quirk only.
- shadcn Button keeps `react-refresh/only-export-components` as an accepted lint *warning* (the canonical upstream file exports `buttonVariants` alongside the component). Lint exits 0 with that one warning.
- Vite dev server uses `strictPort: true` on **1432** (Tauri requirement; HMR fallback 1433). Moved off 1420 specifically because the sibling `requi` repo dev-serves on 1420 - same-port clashes meant a headless screenshot of localhost:1420 could silently capture requi's app instead. Both the vite `server.port` and tauri.conf.json `devUrl` must agree. Also: a `npm run dev` started with `&` in a Bash tool call dies when that call's shell exits - launch it with the tool's `run_in_background` instead so it survives across calls.
- `cmdk` (command palette) calls `Element.prototype.scrollIntoView` on mount to keep the highlighted item visible; jsdom doesn't implement it, so any test rendering a `Command`/`CommandDialog` throws `TypeError: e.scrollIntoView is not a function`. Fix is a one-line no-op polyfill in `src/test/setup.ts` (next to the `ResizeObserver` stub). cmdk filters/highlights rows itself - assert on `role="option"` rows + the `CommandEmpty` text. Also: in an integration test the sidebar list AND the viewport both render the active video's name, so scope name assertions to the viewport region (`within(getByRole("region", { name: /video viewport/i }))`) to avoid "found multiple elements".
- Tauri asset protocol needs BOTH config and a Cargo feature: enabling `app.security.assetProtocol` in `tauri.conf.json` is not enough - the `tauri` crate must also carry `features = ["protocol-asset"]`, or `cargo build` fails with "The `tauri` dependency features ... does not match the allowlist defined under `tauri.conf.json` ... add the `protocol-asset` feature". `convertFileSrc` then resolves local files for `<video src>`.
- jsdom has no real media engine: `HTMLMediaElement.prototype.play/pause/load` throw "Not implemented", so any test that mounts a real `<video>` needs no-op stubs in `src/test/setup.ts` (`play` returns a resolved promise). Also `currentTime`'s setter throws - back it with a plain field via `Object.defineProperty` so seek code (`video.currentTime = sec`) is observable. And `Element.prototype.setPointerCapture/releasePointerCapture/hasPointerCapture` throw - stub them (assign directly; a `if (!...)` guard won't override the throwing built-in) for any pointer-drag UI (the seek slider). Real playback can only be verified manually via `npm start`; tests assert observable state (src attribute, isPlaying-driven icon, aria-* on the slider) and mocked IPC calls, never pixels. The single mockable IPC seam is `@/lib/tauri` - mock that, never the components.
- The `<video>` element must NEVER be conditionally unmounted/remounted, or playback restarts from 0. Bug that bit us: `WorkspaceLayout` returned two different parent trees (a bare `<div>` when the sidebar was hidden vs a `<ResizablePanelGroup>` when shown), so toggling the sidebar remounted `Content` -> the `<video>` -> restart. Fix: one stable tree where the content `<ResizablePanel key="content">` is always mounted and only the sidebar panel/handle are conditionally rendered. Guard this with a test asserting DOM-node identity survives the toggle: `expect(container.querySelector("video")).toBe(before)`. (Also: `react-resizable-panels` in this repo does NOT accept an `order` prop - a stable `key` is enough to preserve the panel across re-renders.)
- Native window fullscreen (Tauri `setFullscreen`) only resizes the OS window - it does NOT hide app chrome. For an immersive view, track an `isFullscreen` flag in `WorkspaceProvider` (set from `toggleFullscreen()`'s returned new state) and gate the sidebar/transport/overlay on it. `toggleFullscreen` returns the new boolean so JS state stays in sync with the window. But that only covers OUR toggle - fullscreen entered/left outside the app (F11, OS Esc, green button) bypasses it and leaves `isFullscreen` stale. Fix: `watchFullscreen(onChange)` subscribes via `getCurrentWindow().onResized` and re-queries `isFullscreen()` on every transition; `Workspace` wires it to `setFullscreen` so the real window state is the single source of truth. The store verb `setFullscreen` is the raw `useState` setter (reference-stable) so the subscribing effect doesn't re-subscribe on every render.
- `getCurrentWindow()` (from `@tauri-apps/api/window`) THROWS outside a Tauri host ("Cannot read properties of undefined (reading 'metadata')") - so it breaks plain-browser `npm run dev` AND any jsdom test that renders the real app without mocking `@/lib/tauri`. Any wrapper using it (e.g. `watchFullscreen`) must `try/catch` and fail safe (return a no-op unlisten), not assume a host. The e2e bootstrap test only mocks `@tauri-apps/api/core`, not `/window`, so it exercises this path.
- Switching the active `<video>` while playing: drive playback from the element's `onLoadedData` (guarded by `isPlaying`) for the freshly-loaded source, and keep the effect's imperative `play()` for resume/pause of the already-loaded element. Calling `play()` synchronously after changing `src` races the load and rejects. Test by firing `fireEvent.loadedData(video)` and asserting `video.paused === false` (back `paused`/`play`/`pause` with a `_paused` field in the jsdom stub).
- UNIVERSAL PLAYBACK = bundled ffmpeg, NOT the webview `<video>` codecs and NOT embedded mpv. The OS webview can't decode every format (macOS WKWebView: no file-based WebM/VP9, no MKV/AVI, no AV1; webkit.org #216652). Embedding mpv (render-API under a transparent webview) was tried and ABANDONED - transparency broke the whole macOS window and the plugin is macOS-untested. The shipping approach: `prepare_media` (Rust, `src-tauri/src/media.rs`) probes codecs via ffprobe; H.264/AAC plays as-is, anything else is transcoded to a **fragmented** MP4 (`-movflags frag_keyframe+empty_moov+default_base_moof`, `h264_videotoolbox` on macOS) and the path is returned the instant the first ~256KB lands. A fragmented MP4 is progressively playable, so `<video>` starts in <1s and streams while ffmpeg races ahead (~15x realtime; AV1 decode is the bottleneck, still far faster than playback). ffmpeg/ffprobe are now BUNDLED static binaries (Tauri sidecars), not system PATH - see the sidecar learning below.
- macOS WKWebView LOSES first-responder after a fullscreen transition (and on launch) - keyboard events stop reaching the webview until the user physically clicks the window. This is tao#208 / tauri#9389, NOT fixable from JS: `getCurrentWindow().setFocus()` focuses the window, not the inner web content view, and `el.focus()` on a DOM node does nothing for it. The ONLY fix is native: call `makeFirstResponder:` on the WKWebView's `NSView` in Rust. See `src-tauri/src/focus.rs` (`focus_webview` command, objc2 - `window.with_webview(|w| msg_send![w.ns_window(), makeFirstResponder: w.inner()])`). Invoke it on every fullscreen transition (in `watchFullscreen`'s onResized) AND every window focus-gain (`onFocusChanged`). Burned ~hours here treating it as DOM/window focus - it is neither.
- KNOWN ENGINE BUG (not app-fixable): on macOS WKWebView/WebKit audio can restart ~0.5-1s after video when resuming H.264 from pause (webkit.org #293544, #221334). In the media pipeline, not the app. Don't chase it in JS.
- Viewport single-click = play/pause, double-click = fullscreen on the SAME element. DON'T debounce the click behind a setTimeout to "wait out" the dblclick - the ~200ms delay is perceptible and feels laggy vs the instant transport button. Instead toggle SYNCHRONOUSLY in `onClick`: a double-click fires `click` twice (toggle + toggle-back = net no change) then `dblclick` -> fullscreen lands on top. Snappy, no stray toggle. The `dblClick -> isPlaying unchanged` test still passes because the two toggles cancel.
- jsdom natively backs `HTMLMediaElement` `volume` / `muted` / `playbackRate` (getters AND setters) - unlike `currentTime`/`play`/`pause`, these need NO stub in `src/test/setup.ts`. So volume/mute/speed tests can assert `video.volume` / `video.muted` / `video.playbackRate` directly on the real element. Float gotcha: stepping volume by `0.05` accumulates IEEE drift (`1 - 0.05 - 0.05 = 0.8999...`), so `changeVolume`/`setVolume` round to 2 decimals (`Math.round(v*100)/100`); playback rate rounds to 1 decimal in `clampRate`. Both keep the on-screen readout/aria clean without a separate format step.
- ffmpeg/ffprobe ship as Tauri **sidecars** (`externalBin` in `tauri.conf.json` = `["binaries/ffmpeg","binaries/ffprobe"]`), resolved in Rust via `app.shell().sidecar("ffmpeg")` (the `tauri-plugin-shell` crate + `shell:allow-execute` capability with `sidecar:true, args:true`). Bare names only - the plugin appends the `-<TARGET_TRIPLE>[.exe]` suffix and picks dev-vs-bundle dir, so DON'T hand-roll `current_exe()` path-joining. `cargo build` HARD-FAILS ("resource path `binaries/ffmpeg-<triple>` doesn't exist") if the binary for the host triple is missing, so `scripts/fetch-ffmpeg.sh` must run before any build. Binaries are gitignored (`src-tauri/binaries/`), fetched + SHA-256-pinned per triple. macOS = GPLv3 (no off-the-shelf LGPL static macOS build exists), Windows = LGPLv3 - see adr.md.
- `tauri-plugin-shell` `.spawn()` (unlike bare `std::process::Command`) force-pipes stdout/stderr into a bounded `channel(1)`; if you don't drain the returned receiver, ffmpeg BLOCKS once the OS pipe buffer fills, stalling a background/streaming encode. Drain it in a detached `async_runtime::spawn` task. Conversely, the detach for streaming WORKS because `CommandChild` has NO `impl Drop` and `kill()` takes `self` by value - dropping the child does not kill ffmpeg, so it keeps encoding the fragmented MP4 after `prepare_media` returns. Read termination via `CommandEvent::Terminated` in the drain task (the plugin owns the child; there's no `try_wait`).
- Radix `DropdownMenu` is modal: while open it sets `aria-hidden` on the trigger, so a test that re-opens the menu between selections only works if the menu CLOSES on each select (the default). Don't `preventDefault` on `onSelect` of a `DropdownMenuCheckboxItem` if tests rely on reopening - keep the close-on-select default and let `onCheckedChange` toggle state. Multi-select is achieved by reopening, not by keeping the menu open.
